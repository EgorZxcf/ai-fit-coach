# План проекта: AI-фитнес-тренер

## Общая идея
Мобильное приложение с AI-тренером: онбординг → генерация плана тренировок → чат с AI-тренером → отслеживание прогресса → подписка. Бэкенд проксирует запросы к AI и хранит данные пользователей — ключи AI API никогда не попадают в само приложение.

## 1. Архитектура (как двигается запрос)

1. **Онбординг** — пользователь указывает цель, уровень, оборудование, ограничения по здоровью → данные уходят на бэкенд.
2. **Генерация плана** — бэкенд собирает промпт (профиль + системные инструкции) → отправляет в AI API → разбирает ответ → сохраняет план в БД → отдаёт на клиент.
3. **Чат с тренером** — сообщение пользователя → бэкенд добавляет контекст (история, профиль, текущий план) → проверяет лимит запросов по тарифу → AI API → ответ сохраняется и возвращается.
4. **Прогресс** — пользователь отмечает выполненные тренировки/вес → пишется в БД → раз в N тренировок бэкенд может предложить адаптацию плана.
5. **Подписка** — покупка через Google Play Billing на клиенте → токен покупки отправляется на бэкенд → бэкенд проверяет его через Google Play Developer API → обновляет статус пользователя.
6. **Уведомления** — локальные пуши о тренировке, без участия бэкенда, чисто Flutter.

```
[Flutter App] <--HTTPS/JSON--> [Backend API] <---> [AI API]
                                      |
                                      v
                                 [PostgreSQL]
```

## 2. Стек технологий

- **Frontend**: Flutter (Dart) — один код под Android/iOS, легко собирается через GitHub Actions без Android Studio на телефоне.
- **Backend**: Node.js + Express (или Python + FastAPI, если другу так удобнее) — простой REST API.
- **БД**: PostgreSQL, можно через Supabase, чтобы не поднимать сервер вручную; либо Firebase Firestore для меньшей возни с инфраструктурой.
- **AI**: Claude API или OpenAI API — вызывается только с бэкенда.
- **Авторизация**: Firebase Auth (быстрее всего) либо своя на JWT.
- **Платежи**: Google Play Billing Library на клиенте + проверка через Google Play Developer API на бэкенде.
- **Хостинг бэкенда**: Render / Railway / Fly.io для старта — бесплатных тарифов хватит на MVP.

## 4. Распределение задач

### Твои задачи (телефон, Termux + GitHub Actions, Flutter)
- Инициализация Flutter-проекта, настройка GitHub Actions для сборки APK
- Экраны: онбординг, чат, план тренировок, прогресс, настройки
- Слой работы с API (HTTP-клиент)
- Локальные уведомления (напоминания о тренировках)
- Интеграция Google Play Billing на клиенте
- Подготовка иконки, скриншотов и текста для Play Store

### Задачи друга (ПК, бэкенд)
- Настройка сервера (Express/FastAPI), структура проекта
- БД: схема пользователей, планов, истории чата, подписок
- Авторизация (Firebase Auth или JWT)
- Прокси к AI API + промпт-инжиниринг для генерации планов и чата
- Rate-limit по тарифам (бесплатный/платный)
- Эндпоинт проверки покупок через Google Play Developer API
- Деплой бэкенда (Render/Railway) + CI

### Общие задачи (сделать вместе и первыми)
- Зафиксировать API-контракт (`docs/api-contract.md`) — какие эндпоинты, какие поля в запросах/ответах. Без этого будете блокировать друг друга.
- Написать системные промпты для AI-тренера и медицинский дисклеймер
- Договориться о структуре БД (поля профиля, формат плана тренировок)

## 5. Этапы

1. **Неделя 1** — планирование: API-контракт, выбор стека, настройка репо и CI
2. **Недели 2-4** — MVP: онбординг, генерация плана, базовый чат, без оплаты
3. **Недели 5-6** — подписка, лимиты запросов, push-уведомления
4. **Неделя 7** — тестирование, полировка UI, подготовка к публикации
5. **Позже** — фото-калькулятор калорий, расширенная аналитика прогресса

## 6. Важно не забыть
- Дисклеймер «не заменяет консультацию врача» — обязателен для категории здоровья в Google Play
- Ключи AI API хранятся только на бэкенде, никогда в коде мобильного приложения
- Политика конфиденциальности нужна для публикации в Play Store
